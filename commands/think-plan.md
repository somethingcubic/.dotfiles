# Plan

## 1. 理解需求

如果 `$ARGUMENTS` 非空，将其作为需求描述的起点。否则一次一个问题逐个深入：

- 要解决什么问题？当前痛点是什么？
- 这份输出主要给谁看？（工程师 / 跨职能 / 混合受众）
- 用户此轮真正需要的产物是什么？（对话结论 / spec / 架构说明 / 阶段计划）
- 最小可验收产物是什么？完成时应该能看到什么？
- 问题的输入、输出、边界、约束、具体难点是什么？
- 这个问题的隐含假设是什么？假设是否成立？
- 更大的背景：为什么要解决这个问题？问题本身是否成立？（XY 问题警觉）
- 预期行为是什么？边界条件？
- 不做什么？（显式排除项）
- 有没有已有方案可以参考？
- 对于特定技术问题，考虑是否可以形式化建模（利用已有求解工具）

UI / 前端视觉任务必须额外锁定：

- 页面类型：app / dashboard / landing / docs / mobile / deck
- 受众与使用情境：谁在什么压力下使用
- 设计系统来源：项目 `DESIGN.md` / 现有 theme-token / 用户提供 / 临时 contract
- 视觉方向：existing / editorial / modern-minimal / warm-soft / tech-utility / brutalist
- 禁用项：渐变、glass、emoji icon、虚构指标、filler copy、无来源图片
- 验收方式：截图 viewport、overflow、状态覆盖、CRAP 原则检查、DESIGN.md adherence

涉及领域概念、业务术语或架构决策时，额外检查：

- 是否已有 `CONTEXT.md` / `CONTEXT-MAP.md` 或领域术语文档；有则优先读取。
- 用户用词是否与已有术语冲突；冲突必须进入 `待决策`，不能沉默选边。
- 是否需要新增领域术语；只记录 domain expert 也会使用的概念，不记录通用编程词。
- 是否存在值得记录的架构决策；只有同时满足“难回退、缺上下文会令人意外、存在真实 trade-off”时才建议 ADR。
- 文档必须 lazy create：只有用户要求沉淀，或本 skill 明确产物包含文档时才创建。

复杂实现任务额外使用 REASONS-lite 自查，但不要把所有小任务强行写成 Canvas：

| 维度 | 检查点 |
|------|--------|
| Requirements | 问题、DoD、Non-goals、验收标准是否明确 |
| Entities | 领域对象、术语、关系是否显式 |
| Approach | 采用什么策略、放弃了什么备选、为什么 |
| Structure | 改哪些文件 / 组件 / 边界，不改哪些 |
| Operations | 步骤是否可执行、可测试、顺序合理 |
| Norms | 项目命名、测试、日志、错误处理等规范 |
| Safeguards | 安全、兼容性、性能、不变量和回退点 |

当后续实现、验证或 review 发现现实与 plan 不一致时，优先更新 plan/spec artifact，再继续实现；不要只在代码上 patch，导致意图记录漂移。

如果发现需求跨多个相对独立的子系统，先建议拆成多个 spec，再继续设计；不要把互不相干的问题硬塞进一个方案。

### 规则

- 每次只问一个问题，等用户回答后再问下一个
- 提出 2-3 个方案并附取舍分析和推荐
- 先锁定读者、产物、验收标准，再拆方案
- 在 spec 中显式区分：`已锁定` / `待决策` / `可自由裁量`
- 遇到技术不确定点，建议用户触发 `/think-research`
- 需要设计/成文架构时，转 `/think-architecture`
- 需要判断结构是否易改、边界是否清晰、模块是否 review-friendly 时，转 `/think-quality`
- **硬门控：设计未经用户批准前，禁止写任何实现代码**

## 2. 选择思维框架

根据问题性质选择合适的思维框架，指导后续方案设计：

| 框架 | 适用场景 | 核心动作 |
|------|---------|---------|
| 问题拆解（横向） | 可并行的子问题 | 按功能模块/时间/空间维度拆分 |
| 问题拆解（纵向） | 流程性问题 | 按步骤/因果链拆解，找出核心瓶颈 |
| 矛盾分析 | 多因素冲突、优先级不清、trade-off 纠缠 | 列出所有对立面 `[A] vs [B]` → 判定主要矛盾（解决它后其他矛盾是否缓解）→ 区分性质（对抗性=必须取舍 / 非对抗性=可协商）→ 集中力量解决主要矛盾，监控次要矛盾是否上升 |
| 拓扑关系梳理 | 复杂系统/多组件交互 | 画出组件关系，分析正向/负向依赖 |
| 类比推理 | 缺少直接经验 | 搜索相同或不同领域的类似问题解法 |
| 模型简化 | 问题过于复杂 | 提取核心矛盾，缩小规模，先解决简化版 |
| 第一性原理 | 现有方案全部失败 | 剥离经验假设，连续问"为什么"回到基本事实 |

不需要每次都显式选择——简单任务直接跳到产出 Spec。复杂任务在 Spec 中标注使用了哪个框架。

## 3. 组织输出层次

无论任务大小，输出都遵循“先让人看懂，再让人看全”的顺序：

1. `TL;DR` — 一句话目标 + 推荐方案
2. `已锁定` — 已确认的目标、边界、约束
3. `待决策` — 仍需用户拍板的问题
4. `边界` — Goals / Non-goals / Constraints
5. `方案` — 方案 A / B / 推荐与取舍
6. `领域语言 / ADR-lite` — 仅在确有术语冲突或关键决策时出现
7. `风险与验证` — 最大风险、验证方式、回退点
8. `实施步骤` — 只展开到足以执行与验证的粒度

不要一上来把所有抽象层、所有 trade-off、所有边角情况平铺出来；优先让读者先获得方向感。

## 4. 产出 Spec

根据任务规模自适应粒度：

### 图示触发条件

出现以下任一情况时，spec 必须附 Mermaid 图：

- 多组件 / 多服务 / 多 agent 交互
- 状态流转、审批流、异步链路
- 仅靠文字容易产生歧义的依赖关系

简单单文件改动、纯局部重命名、单函数级修补不强制画图。

### 小任务（单文件/单函数级）

直接产出 Action 描述级 spec：

```
## TL;DR
[一句话目标 + 推荐方案]

## 已锁定
- ...

## 边界
- Goals
- Non-goals
- Constraints

## 步骤
1. [文件路径] — [操作] — [验证方式]
2. ...
```

### 复杂任务（多文件/多方向）

先产出整体 spec（Action 描述级），再按方向拆分为独立的详细 spec：

```
docs/specs/
├── overview.md          # 整体 spec：目标、架构、各方向概述
├── direction-a.md       # 方向 A 详细 spec（含关键接口/契约/必要代码片段）
├── direction-b.md       # 方向 B 详细 spec（含关键接口/契约/必要代码片段）
└── ...
```

每个方向的详细 spec 包含：

- 主要文件路径和方向性操作
- 关键接口、契约、必要代码片段（仅在消歧需要时给完整代码）
- 验证策略与验收方式
- 读者如何阅读这份 spec（先看什么，再看什么）
- 与其他方向的依赖关系

## 5. 自审

Spec 写完后执行检查清单：

- [ ] 已明确读者、产物、最小可验收产物
- [ ] 已区分 `已锁定` / `待决策` / `可自由裁量`
- [ ] 无占位符（TBD/TODO/"后续完善"）
- [ ] 每步有具体文件路径
- [ ] 每步有验证方式
- [ ] 排除项显式列出
- [ ] 方向间依赖关系无环
- [ ] 垂直切片优先（按功能而非技术层拆分）
- [ ] 复杂交互已补 Mermaid 图（如适用）
- [ ] 改动范围尽量局部，一个需求不无端扩散到很多文件
- [ ] 业务规则已显式化，而不是留给实现时“边写边猜”
- [ ] 边界清晰：核心逻辑与 side effects 分离
- [ ] 标题本身可扫描，读者不必通读全文才知道结论
- [ ] 没有把所有细节一次平铺到压垮读者
- [ ] 术语、命名、接口名在全文保持一致
- [ ] 存在明确验证护栏；必要时先进入 `/dev-tdd`

## 6. 硬约束（与"行为不变""先搜后改"同级，不可绕过）

### Domain Language / ADR-lite

涉及业务概念时，spec 必须使用项目已有术语；发现一词多义或多词同义时，放入 `待决策`。建议沉淀术语时使用以下格式：

```md
**Canonical Term**:
一句话定义这个概念是什么。
_Avoid_: 旧词、歧义词
```

ADR 建议必须克制。只有同时满足以下三条才写入计划：

1. 难回退：以后改回来成本明显。
2. 缺上下文会令人意外：未来读者会问“为什么这样做”。
3. 真实 trade-off：存在合理备选且已经做了取舍。

不满足三条时，不要为了“看起来规范”创建 ADR。

### Premise Collapse（脆弱假设显式化）

每个方案必须显式指认它"最脆弱的假设"——形式固定为：

```
If <X> does not hold, <Y> happens.
```

- `<X>`：这个方案隐含依赖的某个事实（用户行为、外部 API 形状、数据规模、并发量、依赖库行为、团队约定……）
- `<Y>`：这个假设崩溃后会发生什么（功能失效 / 数据损坏 / 性能塌方 / 需要回滚 / 切换方案）

**每份 plan 至少 1 条 premise collapse**；多方案对比时每个候选方案各自给一条。

举例：

- `If 用户量 < 10 万，<内存缓存方案>。If does not hold, 缓存击穿 + DB 雪崩，必须切 Redis。`
- `If 第三方 webhook 重试 ≤ 3 次，<幂等键方案>。If does not hold, 重复消费导致订单重复创建。`

写不出来 = 没真正想清楚方案的边界，回到第 2 节继续推。

### Blocking Ambiguity（矛盾必须先问）

如果两个来源 / 解释 / 文档 / 历史决策之间存在矛盾，**必须先问用户哪个优先**，不允许沉默选边。

触发条件（任一）：

- 用户当前需求与已有代码契约 / 注释 / 设计文档冲突
- 两份来源（issue / 文档 / commit message / 同事意见）给出互斥指令
- 同一术语在不同位置含义不一致
- "之前是 A，但现在的实现是 B"——必须先问用户哪个是事实

**禁止**："我猜用户更倾向 A，就按 A 推方案"——这种"沉默选边"会把矛盾埋进 spec，到实现期或 review 期才爆出来，代价更高。

**做法**：在 spec 的"待决策"区显式列出矛盾点 + 候选项 + 推荐 + 推荐理由，等用户拍板后再继续。

## 7. 用户审批

将 spec 提交用户审批。未批准不进入实现。

## 8. Gotchas

- 不要把 spec 写成“实现草稿”或 TODO 清单；必须写清目标、边界、路径、验证
- 小任务不要强行过度流程化，但也不能跳过关键约束澄清
- 没有文件路径和验证方式的步骤，等于不可执行 spec
- 先写长篇背景、把结论埋在后面，会显著增加认知负担
- “类似上文”“按同样方式处理”属于隐藏占位符，不能用
- 复杂方案先给地图，再给细节；不要让读者自己从碎片里拼全貌
- spec 的价值是减少歧义，不是把所有细节都锁死到无法调整

## 禁止

- 跳过需求理解直接写 spec（"需求很简单"也不行）
- spec 中留占位符或模糊描述
- 用"简化版/v1/先写死"偷偷缩减用户决策的范围
- 把执行计划技能（原子任务、commit 指令）硬塞进 spec 技能
- 没确认读者是谁，就默认用某种术语密度和展开深度
- 没有验证方式的步骤
- 把文件级原子步骤、commit 节奏、逐步执行命令写成 spec 主体

## 关联技能

- 技术不确定时 → 建议用户触发 `/think-research`

- 现有表述太密、太绕、太难读 → `/readable-final-answer`
