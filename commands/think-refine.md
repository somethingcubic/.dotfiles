# Refine

在执行前，先和用户来回澄清，直到"要做什么/不做什么/验收标准"都明确。

## 核心循环

```
1. 理解：复述你理解的需求（一句话 + 关键约束）
2. 提问：列出 1-3 个关键不确定点，每次只问一个
3. 确认：用户回答后更新理解
4. 重复 2-3 直到无需澄清
5. 展示：最小可验收的 plan（不是 spec，是动手清单）
6. 执行或转接 /think-plan（如任务复杂）
```

涉及业务概念时，澄清过程还要检查领域语言：

- 用户用词是否已有项目 canonical term。
- 同一词是否被用来指代多个概念。
- 多个词是否其实指向同一概念。
- 新术语是否是领域专家会使用的概念，而不是通用技术词。

发现术语冲突时，优先问“哪个词作为 canonical term”，并把旧词列为 avoid / alias；复杂到需要成文 spec 时转 `/think-plan`。

## 何时问

在以下任一成立时优先澄清而非开工：

- 需求里有歧义词（"优化一下" / "改好看点" / "加一下认证"）
- 领域术语冲突或一词多义
- 用户提的是方案不是问题（XY 问题警觉）
- 有多种合理解释，选错代价高
- 验收标准不明确

## 何时不问

- 需求已经具体到文件级 + 可验证
- 典型 bug 修复（错误信息已给、复现路径清晰）
- 纯机械操作（改名、格式化、rebase）

## Ambiguity Score（量化停止条件）

不靠"感觉差不多了"判断是否可以放行，用 4 维评分给自己一道闸。每维 0-1，**越低越清晰**：

| 维度 | 含义 | 评分参考 |
|---|---|---|
| Goal Clarity | 目标是否唯一、可被验证 | 0=单一目标 + 可验收；1=目标模糊或多义 |
| Boundary Clarity | 范围 / 不做什么是否明确 | 0=Goals/Non-goals 都列了；1=只有想做没有不做 |
| Constraint Clarity | 约束 / 边界条件是否枚举 | 0=性能/兼容/安全/依赖都列了；1=完全未提 |
| Acceptance Clarity | 验收标准是否可执行 | 0=有具体命令/case/指标；1=只有"看起来不错" |

**总分 = 四维平均**

- ≤ 0.20：放行，可以转 `/think-plan` 或直接动手
- > 0.20：必须继续追问，不允许"差不多了就开工"

### 示例

需求："实现登录"

| 维度 | 评分 | 该追问什么 |
|---|---|---|
| Goal Clarity | 0.6 | 是密码登录 / OAuth / Magic Link / SSO？登录后跳转哪里？ |
| Boundary Clarity | 0.8 | 包不包含注册、找回密码、账号锁定？要不要 2FA？ |
| Constraint Clarity | 0.7 | 现有 user 表结构？session 用 JWT 还是 cookie？依赖哪个 auth 库？ |
| Acceptance Clarity | 0.6 | 怎么算"登录成功"？要测哪些 case（错密码 / 限流 / CSRF）？ |

总分 = (0.6+0.8+0.7+0.6)/4 = **0.675**，远高于 0.20，必须继续追问，**不允许直接开工**。

每追问一轮重新打分，直到 ≤ 0.20 或 3-5 轮无法收敛（按"Gotchas"兜底）。

### 强调

- **评分是给 agent 自检用的工具，不是给用户表演**
- 不需要把分数贴给用户看，也不需要每一维都让用户打分
- agent 自己心里过一遍：哪一维高，下一个问题就追那一维
- 评分高 ≠ 用户能力差，而是输入信号不足，agent 必须主动补足

## 提问规范

- 一次一个问题，不要一次列 5 个让用户疲劳
- 每个问题提供 2-3 个候选选项 + 推荐
- 不要替用户做决定（"我觉得 A 更好，就按 A 做了"）

## 与其他 skill 的边界

- `think-plan` 产出完整 spec，触发门槛更高
- `think-refine` 只把需求澄清到"可以动手"即止，产物是动手清单而非 spec
- 澄清完后如果任务复杂，转 `/think-plan`
- 澄清完后如果任务直接可做，动手

## Gotchas

- 不要把 refine 做成无限循环，3-5 轮无法收敛时直接给出 3 个方案让用户挑
- 用户说 "你看着办" 时不是让你真的随便办，而是让你给推荐 + 兜底方案
- refine 过程本身也要尊重用户时间，不是问得越细越好
