连续失败时按以下等级升级行为约束。每级绑定强制动作，不只是"再试一次"。

## Handoff 接续（与 dev-debug 对齐）

如果上游 `/dev-debug` 已用 Handoff 格式交接进来（连续 2 次假设验证失败触发），不要重新组织信息，直接接同一组表头继续推进：

1. **Symptom** — 症状描述 + 复现命令
2. **Hypotheses Tested** — 已试过哪些假设 + 为什么失败
3. **Evidence Collected** — 已经掌握的事实数据
4. **Ruled Out** — 已被证伪的方向
5. **Unknowns** — 仍不清楚的关键点
6. **Suggested Next Steps** — 建议路径，至少 2 个候选

think-unstuck 的工作是在这份 Handoff 上继续推进：补充 Evidence、扩充 Hypotheses、收敛 Unknowns，直到 Suggested Next Steps 收敛到可执行单一动作或进入"结构化退出"。

非 Handoff 路径（用户手动触发或从其它 skill 进入）按下方压力升级流程走。最终若需要退出，仍用同一组 6 表头，让两个 skill 输出格式一致。

## 压力升级

| 连续失败 | 级别 | 强制动作 |
|---------|------|---------|
| 2 次 | L1 | 切换到**本质不同**的方案（换参数/换函数名不算） |
| 3 次 | L2 | 搜索（WebSearch/Grep）+ 读源码上下文 50 行 + 列 3 个不同假设 + 反转主假设 |
| 4 次 | L3 | 完成下方 7 项检查清单，逐项勾选 |
| 5 次+ | L4 | 强制切换方法论（见下方切换链）；若所有方法论穷尽，输出结构化失败报告 |

## 7 项检查清单（L3 强制）

- [ ] 逐字读完错误信息？
- [ ] 用工具搜索过核心问题？
- [ ] 读过失败位置的上下文（上下 50 行）？
- [ ] 所有假设都用工具确认过？
- [ ] 试过反转主假设？（一直假设问题在 A → 假设问题不在 A）
- [ ] 在最小范围内复现？
- [ ] 换过工具/方法/角度/技术栈？

## 抗合理化

输出以下话术前，先执行对应动作：

| 话术 | 应对 | 触发 |
|------|------|------|
| "超出能力范围" | 方法论走完了吗？穷尽了吗？ | L1 |
| "建议用户手动处理" | 这是你的任务，先穷尽自己的手段 | L3 |
| "可能是环境/版本问题" | 用工具验证了吗？ | L2 |
| "我无法解决" | 7 项清单全完成了吗？ | L4 |
| 反复微调同一处 | 换本质不同的方案 | L1 |
| 空口说"已完成" | 验证命令跑了吗？输出在哪？ | L2 |

## 方法论切换

切换方法论前，先用以下思考框架重新审视问题：

- 问题拆对了吗？（横向拆分 vs 纵向因果链）
- 在解决正确的问题吗？（第一性原理 — 剥离假设回到基本事实）
- 有没有类似问题的已知解法？（类比推理 — 搜索同类问题）
- 问题能不能简化？（模型简化 — 先解决缩小版）

5 种方法论代表不同的解题方向。按任务类型选择起始方法论，失败时按切换链跳转到下一个。

| 方法论 | 核心动作 | 适用场景 |
|--------|---------|---------|
| RCA 根因分析 | 5-Why 逐层追问根因 + 反转假设自攻击 | Debug、修 bug |
| The Algorithm | 质疑需求→删除→简化→加速→自动化（严格按序） | 新功能、方案设计 |
| 搜索优先 | 强制搜索→读原文→再判断，禁止凭记忆下结论 | 调研、未知问题 |
| Working Backwards | 从期望输出倒推→先定义验收标准→再实现 | 架构决策 |
| 数据驱动 | 测量 before/after→A/B 对比→数据说了算 | 性能优化 |

### 失败切换链

按失败模式选择切换路径，已尝试的方法论不重复：

- **原地打转**（反复改参数不改思路）→ The Algorithm → 搜索优先 → RCA
- **放弃/推锅**（"建议手动""超出范围"）→ RCA → The Algorithm
- **没搜就猜**（凭记忆下结论不验证）→ 搜索优先 → Working Backwards → 数据驱动

切换前三问（防无效切换）：
1. 当前方法论的核心步骤都走了吗？没走完不切。
2. 失败是方法论不对还是执行不到位？执行问题不切。
3. 新方法论能解决当前失败模式吗？不能别切。

## 结构化退出

7 项清单全完成且仍无解时，输出失败报告：

1. **已验证事实** — 用工具确认过的事实
2. **已排除可能** — 每条附排除证据
3. **缩小的范围** — 问题被定位到哪个区间
4. **推荐下一步** — 用户可以采取的行动

## Gotchas

- `think-unstuck` 不是换个说法继续碰运气，而是强制切换方法论
- 没做完当前等级的强制动作，就不要声称“已经试过很多办法”
- 连续失败时最常见的问题不是能力不够，而是一直在重复同一种错误路径
- 退出报告必须基于证据，不要把情绪化挫败感写成结论
