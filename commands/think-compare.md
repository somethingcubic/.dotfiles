# Compare

用于把“选哪个”问题从直觉判断变成维度化比较。重点不是先站队，而是先问：这个问题到底应该比较哪些维度？

## 1. 明确比较对象

解析 `$ARGUMENTS`，先列出候选方案：

- 如果候选方案明确，直接使用用户给出的方案。
- 如果候选方案不完整，补充 1-2 个合理备选，并标注 `[推断]`。
- 如果问题本身不是选择题，先改写成可比较的问题，再继续。
- 如果关键信息缺失到无法比较，先问最少量澄清问题。

## 2. 先定义比较维度

不要直接比较方案。先根据问题性质列出 4-8 个维度，并说明每个维度为什么重要。

常用维度：

| 问题类型 | 可比较维度 |
|---|---|
| 技术选型 | 现有生态契合度、复杂度、可维护性、性能、可测试性、迁移成本、长期风险 |
| 实施路径 | 交付速度、风险、回滚难度、验证成本、影响面、团队熟悉度 |
| 架构方案 | 边界清晰度、扩展性、数据一致性、故障隔离、可观测性、运维成本 |
| 产品/策略 | 用户价值、机会成本、可逆性、学习价值、执行成本、时间敏感性 |

维度必须来自当前问题；不要机械套模板。每个维度应能区分方案，否则删除。

## 3. 逐维比较

输出固定表格：

```md
| 维度 | 为什么重要 | 方案 A | 方案 B | 方案 C | 胜出者 |
|---|---|---|---|---|---|
| ... | ... | 优势/劣势 | 优势/劣势 | 优势/劣势 | A / B / C / 平手 |
```

规则：

- 每格写可判断的差异，不写空泛形容词。
- 胜出者可以是“平手”，但必须说明为什么没有明显差距。
- 如果某维度依赖未验证前提，标注 `[未验证]`。
- 重要维度可以加权，但必须显式说明权重理由。

## 4. 给出选择

结尾必须给出明确裁决：

```md
## 结论

推荐：方案 X / 平手

理由：
1. ...
2. ...
3. ...

适用前提：
- ...

如果前提变化：
- 当 ... 时，改选方案 Y
```

如果结果是平手，说明：

- 哪些维度打平；
- 需要补充哪一个信息才能打破平手；
- 如果必须现在选，默认选哪个更稳。

## Gotchas

- 不要先有结论再倒推维度。
- 不要只列优缺点清单；必须逐维比较并给胜出者。
- 不要把所有维度等权处理；关键约束应高于次要偏好。
- 不要用“看情况”逃避裁决；可以给条件化裁决或平手裁决。
- 不要把事实、推断、偏好混在一起；未验证内容必须标注。
