# Map

## Decision Principles

- `think-map` 优化的是对陌生仓库的整体方向感，不是穷尽阅读每个文件。
- 地图要回答“入口在哪里、约定是什么、验证怎么跑、风险集中在哪”，因为这些信息决定后续 plan、debug 和 review 是否会走偏。
- 当已能支持下一步决策时停止；继续 repo-wide 搜索只会增加噪音，除非发现入口、依赖或验证命令仍不清楚。
- 未确认的信息必须标成未知或待查，不能为了地图完整感把猜测写成事实。

## 1. 扫描

对 `$ARGUMENTS` 指定的目录（留空则为项目根目录）执行结构化分析：

- 先读根 `README`、`AGENTS.md`、主要配置文件，建立全局约束与入口认知

- 目录树概览（忽略 node_modules/.git 等）
- 识别主要语言和框架
- 找到入口文件和配置文件
- 识别构建/测试/lint 命令（package.json/Makefile/pyproject.toml 等）

## 2. 分析维度

| 维度 | 产出 |
|------|------|
| 技术栈 | 语言、框架、核心依赖及版本 |
| 架构 | 模块划分、数据流向、分层结构 |
| 约定 | 命名规则、目录组织、代码风格 |
| 测试 | 测试框架、覆盖范围、运行命令 |
| 依赖 | 模块间依赖关系、外部服务集成 |
| 关注点 | 技术债、已知问题、需要注意的区域 |

## 3. 输出

产出简洁的代码地图（直接输出到对话中，不创建文件）：

```
## 技术栈
[语言/框架/核心依赖]

## 架构
[模块划分和数据流]

## 约定
[命名/目录/风格规则]

## 构建与测试
[关键命令]

## 关注点
[技术债/风险区域]
```

## 规则

- 只读分析，不修改任何文件
- 优先用 Glob/Grep/Read 工具，不凭猜测
- 大型项目可派发子 agent 并行分析不同模块
- 关注点要具体（指出文件和行号），不泛泛而谈

## Gotchas

- 不要只列目录树；代码地图的重点是入口、约定、依赖和风险点
- 没读 README/AGENTS/配置就下结论，通常会误判项目约束
- 结论尽量带 `file:line` 或至少具体文件路径，避免空泛描述
- map 是后续 `/think-plan`、`/dev-debug`、`/guard-review` 的地基，不要把未知项伪装成已确认事实
