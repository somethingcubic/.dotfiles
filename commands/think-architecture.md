# Architecture

## 1. 先锁问题，不急着画图

先明确这次架构工作的边界：

- 这是 **系统/服务架构**，还是 **仓库/代码架构**？
- 主要读者是谁？默认按工程师处理，但要确认是否有跨职能读者
- 输出要放在对话里，还是直接形成架构文档？
- 这次是要做新设计、评审现有设计，还是重写已有架构说明？
- 目标、非目标、约束、风险预算是什么？
- 已锁定哪些决策？哪些还待定？

如果问题本身还不清晰，先收敛问题，再谈方案。
如果技术选型不确定，先走 `/think-research`。
如果后续要进入具体 spec 或阶段计划，完成架构判断后转 `/think-plan`。

## 2. 选择视角，而不是一次讲完所有层

按问题选择合适的观察透镜：

| 透镜 | 适用问题 | 关注点 |
|------|---------|-------|
| C4 风格分层 | 需要逐层拆系统、服务、容器、组件、代码结构 | “系统由什么组成” |
| 4+1 风格关注点 | 不同角色关心点不同 | “不同读者分别最关心什么” |
| ADR-lite | 存在关键取舍、需要记录为什么这么选 | “为什么这样做而不是那样做” |
| arc42 风格章节 | 需要成文且避免遗漏关键版块 | “文档是否完整、可导航” |

不要把模板当答案。模板只是组织视角，核心是让读者逐层理解。

### 领域语言与 ADR-lite

架构说明必须尊重项目已有领域语言：

- 如果存在 `CONTEXT.md` / `CONTEXT-MAP.md`，先用它确定术语。
- 如果同一术语在不同位置含义不同，列入 `Open Questions`，不要自行选边。
- 新术语只记录领域专家也会使用的概念，避免把通用技术词塞进领域语言。

ADR-lite 只在三条同时成立时建议创建：难回退、缺上下文会令人意外、存在真实 trade-off。ADR 可以只有 1-3 句，重点是“做了什么决定，为什么”，不是填模板。

## 3. 输出顺序：先地图，再下钻

默认按下面顺序输出，除非用户明确要求别的格式：

1. `TL;DR` — 这套架构解决什么问题，推荐结构是什么
2. `Reader Guide` — 这份内容怎么读，先看哪部分
3. `Context` — 系统边界、外部依赖、上下游
4. `Containers or Repo Modules` — 主体拆分与职责
5. `Critical Flows` — 关键链路、数据流、状态流
6. `Key Decisions` — 关键取舍、放弃了什么、为什么
7. `Risks / Non-goals` — 已知风险、明确不做什么
8. `Open Questions` — 仍待决策的问题

如果存在领域术语冲突，在 `Open Questions` 中显式标出候选术语、冲突来源和推荐项；若建议 ADR，在 `Key Decisions` 中写明三条触发条件是否成立。

每一层最多列 5–7 个元素；超出就继续分组，不要平铺。

## 4. 图示规则

图只在能降低歧义时才画：

- 多组件/多服务交互
- 关键异步链路
- 状态流转
- 单靠文字很难建立全局地图

默认优先：

1. 1 张总览图
2. 1 张关键链路图

每张图后都要跟一句文字总结：**这张图想让读者看懂什么**。

## 5. 文档模式

### 对话模式

适合先收敛方向：

```md
## TL;DR
## 已锁定
## 待决策
## 推荐结构
## 关键取舍
## 下一步
```

### 成文模式

适合直接产出架构说明：

```md
## TL;DR
## Reader Guide
## Context
## Containers or Repo Modules
## Critical Flows
## Key Decisions
## Risks / Non-goals
## Open Questions
```

系统/服务架构 与 仓库/代码架构 都沿用这个骨架，只替换主体内容：

- **系统/服务架构**：强调边界、服务职责、数据流、部署约束
- **仓库/代码架构**：强调目录/模块边界、依赖关系、扩展点、测试护栏

## 6. 认知负担规则

- 先给一屏摘要，不要把总览藏在后面
- 先写“是什么/为什么”，再写“怎么实现”
- 标题要能独立表达小结论
- 术语首次出现时解释
- 不把 trade-off、边角条件、实施细节同时灌给读者
- 遇到密文、大段黑话、绕句时，转 `/readable-final-answer` 或套用其规则

## 7. 自检

- [ ] 读者是否能先看到一张地图，再逐层下钻？
- [ ] 是否明确了 goals / non-goals / constraints？
- [ ] 是否把关键决策与理由写出来了？
- [ ] 是否区分了“现状”“推荐方案”“未决问题”？
- [ ] 图是否真的降低了歧义，而不是装饰？
- [ ] 每层信息量是否可扫描，而不是全文信息平铺？

## 8. Gotchas

- 不要把“架构文档”写成“实现细节流水账”
- 不要一开始就深入组件级细节，导致读者没有全局地图
- 不要只有框图没有责任描述；也不要只有责任描述没有关键链路
- 不要把“未来可能会做”写成“已经决定要做”
- 不要在这里展开文件级改动清单、原子步骤、执行命令；这些属于 `/think-plan`
