# Postmortem

基于当前对话中的问题排查过程，进行结构化复盘。

## 执行步骤

### 1. 还原时间线

从当前对话中提取：
- **发现**：问题怎么暴露的
- **尝试**：试了哪些方案，为什么失败
- **解决**：最终怎么修的

### 2. 根因分析

区分：
- **表面原因**：直接触发问题的因素
- **根本原因**：为什么会出现这个表面原因
- **系统性因素**：流程/规范/工具层面有什么缺失

### 3. 提炼教训

提炼 1-3 条可复用的教训，每条包含：
- 规则本身
- 适用场景
- 违反后果

### 4. 写入存档

写入 `~/.claude/postmortems/YYYY-MM-DD-<topic>.md`，格式：

```markdown
# Postmortem: <标题>

**日期**: YYYY-MM-DD
**项目**: <项目名称和路径>
**严重程度**: 低/中/高

## 时间线
...

## 根因分析
- 表面原因: ...
- 根本原因: ...
- 系统性因素: ...

## 教训
1. ...

## 行动项
- [ ] ...
```

### 5. 判断是否需要全局更新

检查教训是否应该：
- 写入 auto memory（feedback 类型）→ 给出建议，等用户确认
- 更新 `~/.claude/CLAUDE.md` 或 `~/.claude/rules/` → 给出具体修改建议，等用户确认
- 仅存档不扩散 → 说明理由

**不自作主张修改全局配置，只给建议让用户确认。**

## 规则

1. 如果当前对话没有明显的问题排查过程，提示用户描述遇到的问题
2. 根因分析必须有证据支撑，标注 [推断] 或 [局部推断]（遵循 Truth Directive）
3. 教训要具体可执行，不要泛泛的"以后要注意"
4. `$ARGUMENTS` 非空时作为问题简述，辅助定位对话中的相关内容
