# Survey

定位：**开放主题的资料汇总与综述**。允许"无明确结论"，但必须说明现状、流派、分歧、置信度。如需在已知方案间做决策，转 `/think-research`。

## 1. 确定主题与边界

解析 `$ARGUMENTS`：

- 主题词（"LLM 推理优化现状"）
- URL 列表 / 文件路径 / PDF 路径
- 二者混合（"基于这几个链接调研 X"）

### 边界澄清（必要时反问用户）

| 维度 | 选项示例 |
|---|---|
| 范围 | 领域 / 时间窗（最近 N 个月）/ 语言（中/英） |
| 深度 | 10 分钟概览 / 1 小时深读 / 多轮迭代 |
| 受众 | 自用速查 / 团队分享 / 决策输入 |

边界不清时优先 `/think-refine`，不要带着模糊主题硬调研。

## 2. 来源策略（三层取证）

| 层 | 工具 | 用途 |
|---|---|---|
| 外部网络 | WebSearch + `/web-read` | 主流做法、最新进展、官方文档 |
| 本仓库 | Grep / Glob / Read | 当前项目是否已有相关代码、依赖、历史决策 |
| 用户提供 | Read（文件路径）/ `/web-read`（URL/PDF） | 用户显式给的素材 |

### 来源优先级

1. 官方文档 / 论文 / 官方仓库
2. 一线维护者 issue / RFC / release notes / ADR
3. 高质量综述 / 教程 / 实战报告
4. 社区文章

冲突时：高优先级胜出，并在"关键分歧"节显式标记冲突点，不擅自调和。

## 3. 大主题判断（决定是否并行）

| 信号 | 主流程 | 派发只读子任务 |
|---|---|---|
| ≥ 3 个并列子方向 | | ✅ |
| ≥ 10 个候选来源 | | ✅ |
| 主题边界清晰、来源 < 10 | ✅ | |

### 子任务委派契约（仅大主题）

```
Goal: 调研 <子方向> 在 <时间窗> 内的 <K> 个高质量来源
Constraints:
  - 只读，不修改文件
  - 按"来源优先级"取证
  - 输出固定格式卡片集，不要自由叙述
Steps:
  1. WebSearch 或读用户给定 URL/文件
  2. 抽取要点（≤ 3 句/条）
  3. 标注来源类型与置信度
Return（固定格式卡片集）:
  | 来源 | 类型 | 要点 | 证据片段 | 置信度 |
```

主流程负责汇总、去重、归类、写综述。子任务卡片集是证据，不是结论。

## 4. 收集与归类

- 每条素材 → 要点（≤ 3 句）+ 来源链接 + 置信度（确认 / 推断 / 未验证）
- 按维度归类（按主题选最贴切的 2-3 个）：
  - **流派 / 路线**（不同技术方向）
  - **时间线**（演化路径）
  - **应用场景**（按使用场景切）
  - **角色 / 玩家**（按公司、社区、个人）
- 去重：相同观点不同来源 → 合并并列出全部来源
- 分歧：不同观点显式列出，不要"平均化"

## 5. 输出综述（固定结构）

```markdown
## Survey: <主题>

### 摘要
- 范围：<领域 / 时间窗 / 语言>
- 来源数：<N>（官方=<x> / 一线=<y> / 综述=<z> / 社区=<w>）
- 是否有共识：<是 / 部分 / 否>

### 背景
为什么这个主题值得调研，问题起源是什么。

### 现状
当前主流做法、关键玩家、活跃程度。

### 流派 / 路线对比
| 路线 | 代表项目/作者 | 核心思路 | 适用场景 | 局限 | 置信度 |
|---|---|---|---|---|---|

### 时间线（可选，仅当主题演化重要时）
| 时间 | 里程碑 | 影响 |

### 关键分歧
- 分歧点 1：A 派认为 ...（来源）；B 派认为 ...（来源）
- ...

### 与本仓库的关系（仅当本仓库相关时）
- 现有依赖 / 已有实现 / 历史决策 / 相关文件路径

### 结论
- 共识：<列出>
- 仍开放的问题：<列出>
- 如需进一步决策 → 转 `/think-research`
- 如需拆 spec → 转 `/think-plan`

### 参考
按层级列出：
- 官方：...
- 一线：...
- 综述：...
- 社区：...

### 置信度自评
- 来源覆盖度：High / Med / Low
- 时效性：High / Med / Low
- 共识强度：High / Med / Low
```

## 6. 中立性与硬约束

- 每条结论标置信度（确认 / 推断 / 未验证），不要把推断当事实
- 矛盾来源显式列出，不擅自调和
- 综述允许"无明确结论"，但必须在"结论"节说明**为何无结论**
- 不脱离当前仓库上下文谈"最佳实践"
- 不把社区博客 / 营销文 当规范；优先官方与一线
- 子任务卡片集是证据不是结论；最终归类与综述由主流程负责

## Gotchas

- 不要用 survey 替代 research：要给推荐时走 `/think-research`
- 不要在主题模糊时硬调研，先 `/think-refine`
- 不要把 WebSearch 搜到的前几条直接当"业界共识"，要按来源优先级筛
- 综述里"无结论"是合法输出；硬凑结论比没结论更糟
- 大主题不派发子任务，主流程容易上下文爆炸；小主题派发子任务是浪费

## 禁止

- 把社区贴 / 营销文当官方来源
- 不给置信度的"事实陈述"
- 把分歧观点自行调和成"折中结论"
- 综述结尾给具体技术选型推荐（越界，转 research）
- 跳过"与本仓库的关系"节（如确实无关，明示"与本仓库无直接关系"，不能省略）

## 关联技能

- 在已知方案间选 / 必须给推荐 → `/think-research`
- 在已知候选间按维度打分 → `/think-compare`
- 抓取单个 URL / PDF → `/web-read`（survey 内部也会调用）
- 主题模糊不清 → `/think-refine`
- 综述完成后要拆 spec → `/think-plan`
- 把研究结果沉淀为规则/模板 → `/assist-learn`
- 综述涉及陌生仓库 → `/think-map`
