# Tech Spec Writer

根据产品需求文档(PRD)生成结构化的技术方案。

## 工作流程

### Step 1: 阅读 PRD

1. 获取用户提供的 PRD 文档路径或内容
2. 提取关键信息：
   - 功能需求列表
   - 非功能需求（性能、安全等）
   - 验收标准
   - 不做什么（边界）

### Step 2: 理解与澄清

针对 PRD 中不明确或有疑问的点，与用户讨论：
- 技术约束：是否有指定的技术栈、框架版本？
- 性能要求：QPS、延迟、数据量级？
- 依赖系统：需要对接哪些外部服务？
- 数据存储：新增表还是复用现有表？
- 部署方式：独立服务还是集成到现有服务？

**注意**：问题要聚焦，每轮不超过 5 个问题。

### Step 3: 代码库分析

在设计前，先分析现有代码库：
1. 识别可复用的模块、函数、模式
2. 了解现有的目录结构和代码规范
3. 找出需要重构或可提取复用的代码
4. 确认依赖库和版本

### Step 4: 设计技术方案

按以下模板结构生成文档：

```markdown
# {项目名称} 技术方案

## 1. 概述
- 背景（简述 PRD 核心目标）
- 技术选型理由

## 2. 架构设计
- 整体架构图（文字描述或 ASCII）
- 模块划分
- 数据流

## 3. 详细设计

### 3.1 目录结构
- 新增文件/目录清单

### 3.2 数据模型
- 新增/修改的数据表结构
- Go struct 定义

### 3.3 接口设计
- API 路由列表
- 请求/响应格式

### 3.4 核心逻辑
- 关键流程伪代码或流程图
- 状态机设计（如有）

### 3.5 复用与重构
- 可复用的现有代码
- 建议重构的代码

## 4. 非功能设计
- 性能优化策略
- 错误处理
- 日志与监控

## 5. 实现计划
- 任务拆分
- 优先级排序
- 预估工时（可选）

## 6. 待确认/TODO
- 需要后续补充的内容
```

### Step 5: 讨论与迭代

向用户展示技术方案草稿，询问：
- 架构设计是否合理？
- 是否有遗漏的技术点？
- 复用/重构建议是否可行？
- 实现优先级是否正确？

根据反馈迭代修改，直到用户确认。

### Step 6: 输出终稿

将最终技术方案保存到文件：
- 默认路径: `{project}/docs/TECH_SPEC_{项目名}.md`
- 用户可指定其他路径

## 核心约束

1. **遵循项目规范**: 阅读项目的 CLAUDE.md 或 README，遵循已有的编码规范和架构风格
2. **优先复用**: 优先使用项目中已有的库、模式、工具函数，不重复造轮子
3. **简单实用**: 代码设计要简单、高效、易维护，不炫技，不过度设计
4. **聚焦 PRD**: 只设计 PRD 中明确要求的功能，不自行扩展
5. **指出重构点**: 如发现可优化或需重构的代码，在方案中明确指出
6. **技术债务透明**: 如有 TODO 或临时方案，必须标注
7. **流程清晰**：如需要绘制流程图等请使用mermaid绘制
