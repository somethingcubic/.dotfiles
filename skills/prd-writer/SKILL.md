# PRD Writer

帮助用户将口语化的需求描述转化为结构化的产品需求文档。

## 工作流程

### Step 1: 信息收集

从用户输入中提取关键信息：
- 项目名称
- 核心目标（为什么要做）
- 目标用户（谁来用）
- 主要功能点

### Step 2: 澄清问题

针对缺失或模糊的信息，提出 **不超过 5 个** 聚焦问题：

| 问题类型 | 示例 |
|---------|------|
| 用户角色 | 谁来用？不同角色的权限差异？ |
| 功能边界 | 必须做什么？明确不做什么？ |
| 数据流 | 输入输出是什么？数据从哪来、到哪去？ |
| 异常场景 | 失败/边界情况如何处理？ |
| 验收标准 | 怎么算做完？可量化的指标？ |

**注意**：问题要收敛，避免过度发散。只问对撰写 PRD 必要的问题。

### Step 3: 生成 PRD 草稿

按以下模板结构生成文档：

```markdown
# {项目名称} PRD

## 1. 背景与目标
- 为什么要做？解决什么问题？
- 成功标准是什么？

## 2. 用户角色
- 谁会使用？不同角色的核心诉求

## 3. 功能需求
### 3.1 {功能模块A}
- 用户故事: 作为{角色}，我希望{做什么}，以便{获得什么价值}
- 具体需求点（编号列表）
- 边界情况/异常处理

### 3.2 {功能模块B}
...

## 4. 非功能需求（可选）
- 性能、安全、兼容性等约束

## 5. 不做什么
- 明确排除的功能，避免范围蔓延

## 6. 验收标准
- 可量化的完成定义
```

### Step 4: 确认与修改

向用户展示 PRD 草稿，询问：
- 是否有遗漏的需求点？
- 是否有需要调整的描述？
- 是否有需要补充的边界情况？

根据反馈迭代修改，直到用户确认。

### Step 5: 输出文件

将最终 PRD 保存到文件：
- 默认路径: `~/samuel/docs/prd/{项目名}.md`
- 用户可指定其他路径

## 核心约束

1. **不丢细节**: 用户提到的任何点都必须体现在 PRD 中
2. **不过度设计**: 只写用户明确表达的需求，不臆测延伸功能
3. **中立描述**: 只描述"要什么"，不涉及"怎么做"（技术方案由其他 skill 负责）
4. **问题收敛**: 每轮最多 5 个问题，聚焦核心信息
5. **中文撰写**: PRD 全文使用中文
6. **流程清晰**：如需绘制流程图等，请使用mermaid绘制

## 示例对话

**用户**: 我想做一个邮件模板管理功能

**AI**: 我来帮你梳理需求。先确认几个问题：
1. 谁会使用模板管理？普通用户还是管理员？
2. 模板需要支持哪些变量/占位符？
3. 是否需要模板分类或分组？
4. 模板创建后是否允许修改/删除？

**用户**: 普通用户都能用，支持 {{name}} 这种变量，不用分类，可以改可以删

**AI**: 明白了，我来生成 PRD 草稿...
